# AI-Native — Repository Setup

This `ai-native/` directory was auto-initialized by the `ai-process-core` skill.

## What's Here

| Path | Purpose | Editable? |
|------|---------|-----------|
| `.ai-process.binding.yaml` | Active binding config — the file the skill reads | Yes — configure per repo |
| `.ai-process.binding.annotated.yaml` | Same structure with full field-by-field comments | Reference only |
| `rules/risk-overrides.yaml` | Repository risk protection rules (protected/denied paths) | Yes — add your paths |
| `docs/ai-index/global-manifest.yaml` | Knowledge routing index — generated by bootstrap | **Only `custom_refs` section** |
| `docs/ai-index/flows/` | Business flow docs — generated by bootstrap | No — auto-generated |
| `docs/ai-index/modules/` | Module structure docs — generated by bootstrap | No — auto-generated |

| `runtime/ai-process/` | Live run state, events, artifacts — generated at runtime | No — managed by scripts |

## First Steps After Init

### 0. Install ccusage (one-time, recommended)

The process tracks token usage per run. For this to work, install `@ccusage/codex` globally:

```bash
npm install -g @ccusage/codex --registry https://registry.npmjs.org
```

This is a one-time step per machine. Without it the process still works but token metrics
(input/output/total tokens, cost) will show 0 in dashboards.

> **Why `--registry`?** Corporate npm registries (e.g. Artifactory) don't mirror this package.
> The flag routes this specific install to the public npm registry.

### 1. Review the binding

Open `.ai-process.binding.annotated.yaml` and read the comments. Then edit `.ai-process.binding.yaml`
to match your team's preferences.

Key fields to configure:

| Field | Default | Change if... |
|-------|---------|-------------|
| `process_core.host` | `codex` | You're using Cursor → set to `cursor` |
| `process_core.development_protocol` | `tdd` | Your team uses DDD or no protocol → `ddd` or `none` |
| `complexity_routing.enabled` | `true` | You always want full flow regardless of request size → set to `false` |
| `stage_review_policy.verify.require_human_approval_before_advance` | `true` | Personal repo / no review needed → `false` |
| `commit_policy.require_commit_for_approval` | `false` | Keep as `false` — required when `mode: none` |
| `validation_entrypoints.verify_checks` | `[]` | Add your real test commands here |

### 2. Configure risk protection

Open `rules/risk-overrides.yaml` and add any paths that should require approval or be denied
for AI writes. Examples:

```yaml
protected_paths:
  - "**/.env*"
  - "ops/prod/**"
  - "db/migrations/**"
```

### 3. Add your team's custom knowledge

The `docs/ai-index/custom/` directory is for knowledge the AI cannot discover automatically.
See the [Custom Knowledge Base Guide](#custom-knowledge-base) below.

### 4. Knowledge bootstrap

The `flows/` and `modules/` knowledge index is empty. The `ai-process-core` skill will run a full
repository knowledge bootstrap automatically on the next session start.

## Custom Knowledge Base

The AI process automatically scans your code to build `flows/` and `modules/` docs. But it cannot
discover your team's history: past decisions, known traps, naming conventions, and abandoned approaches.

That is what `docs/ai-index/custom/` is for.

### Directory Layout

```
docs/ai-index/custom/
  README.md            ← This guide (inside custom/)
  architecture/        ← Historical design decisions, architectural trade-offs
  gotchas/             ← Known traps and non-obvious failure modes
  conventions/         ← Team agreements on naming, patterns, constraints
  deprecated/          ← Approaches tried and abandoned, with reasons
```

### How to Add a Knowledge File

**Step 1 — Write the file**

Create a Markdown file in the appropriate subdirectory:

```
docs/ai-index/custom/gotchas/webhook-retry.md
```

Use this format for best AI readability:

```markdown
# Webhook 重试的幂等性陷阱

## 适用范围
所有使用 WebhookDispatcher 发送外部通知的场景。

## 背景
2024-Q3 曾因重试逻辑缺少幂等性检查，导致同一事件被重复处理三次，
触发了下游订单系统重复扣款问题。

## 核心结论
- 所有 webhook handler 必须在处理前检查 event_id 是否已处理（幂等键）
- 超时配置使用 WEBHOOK_TIMEOUT_MS 环境变量，默认 5000ms，不要硬编码
- 重试间隔必须使用指数退避，不能用固定间隔

## 细节
幂等键存储在 Redis，TTL 24h。见 src/webhook/idempotency.ts。

## 不适用场景
内部事件总线（EventBus）已有独立的幂等机制，不受此规则约束。
```

**Step 2 — Register in global-manifest.yaml**

Open `docs/ai-index/global-manifest.yaml` and add an entry to `custom_refs`:

```yaml
custom_refs:
  - path: docs/ai-index/custom/gotchas/webhook-retry.md
    scope: [webhook, retry, dispatcher, async, notification]
    summary: "Webhook 重试的幂等性陷阱：必须做幂等键检查，使用指数退避"
```

- `path`: relative to the repo's `ai-native/` directory
- `scope`: keywords that trigger loading this file during Clarify. Be specific — vague keywords
  cause unrelated context to be loaded. Use module names, flow names, or domain terms.
- `summary`: one sentence shown to the AI before it decides whether to load the full file.

**Step 3 — Done**

The next time you make a request that touches webhook-related code, the Clarify stage will
automatically load this file into the context bundle, and the AI will apply the constraints
when writing spec and implement.

### When the AI Suggests a New Custom Entry

At the end of a run, during the **Learn stage**, the AI may observe:
> "This run discovered a new constraint on X. Consider adding a gotcha entry to `custom/gotchas/`."

This is a soft suggestion — you decide whether to act on it. If you do, follow Steps 1-2 above.

### Maintenance Checklist

- [ ] After any significant incident or near-miss → add to `gotchas/`
- [ ] After a major architectural decision → add to `architecture/`
- [ ] After deprecating an approach → add to `deprecated/` (do NOT delete the old code entry,
  just explain why it was abandoned)
- [ ] When a file becomes stale (underlying code changed) → update or delete the file AND
  remove/update its `custom_refs` entry in `global-manifest.yaml`

## Notes

- Do not edit `docs/ai-index/flows/` or `docs/ai-index/modules/` manually — they are
  generated by the bootstrap process and will be overwritten.
- Do not edit `runtime/ai-process/` files manually — managed by orchestration scripts.
- `docs/ai-index/custom/` is the **only** part of `docs/ai-index/` that is entirely yours.
- Add `ai-native/runtime/` to `.gitignore` if you don't want to commit run artifacts.
- The `custom_refs` section in `global-manifest.yaml` survives bootstrap reruns — the bootstrap
  only touches `modules:` and `flows:`, never `custom_refs`.
